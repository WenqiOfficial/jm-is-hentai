# 核心代码与架构审查规范 (Code Review Standards)

本规范旨在确保项目代码达到最高水准：极简、高内聚、低耦合、绝对鲁棒。

## 0. 核心哲学 (Core Philosophy)
- **KISS (Keep It Simple, Stupid)**: 抵制过度设计。寻找最直接、最易读的实现路径。
- **YAGNI (You Aren't Gonna Need It)**: 砍掉一切“未来可能用到”的预留代码、接口和无用变量。
- **DRY (Don't Repeat Yourself)**: 绝不容忍复制粘贴的代码。发现重复，立即抽象。
- **奥卡姆剃刀 (Occam's Razor)**: 如无必要，勿增实体。能用原生解决的，不引入依赖；能删减的模块，坚决移除。

## 1. 后端与逻辑层准则 (Logic & Architecture)
- **纯函数优先**: 尽可能减少副作用，保持状态的单向流转。
- **防御性编程 (Defensive Programming)**: 永远不要信任外部输入。边界条件、异常捕获、空值处理必须严密，保证程序不崩溃（鲁棒性）。
- **时间与空间复杂度**: 评估关键路径算法，指出不必要的嵌套循环或内存泄漏风险。
- **错误处理**: 拒绝宽泛的 `catch (e)` 并在控制台打印了事。错误必须被妥善分类、抛出或降级处理。

## 2. 前端与界面准则 (Frontend & UI Standards)
- **原子化与组件复用 (Atomic Design)**: 拆解庞大的组件，确保每个组件只做一件事（单一职责原则）。
- **样式极致精简**: 剔除死代码 (Dead CSS)、重复的样式类。优先使用现代 CSS 特性 (如 Flex/Grid, CSS Variables) 替代臃肿的布局脚本。
- **性能与渲染 (Performance)**: 严查不必要的重渲染 (Re-renders)、未优化的巨型列表、未懒加载的资源。
- **无障碍与语义化 (A11y & Semantic)**: 检查 HTML 标签使用的正确性，拒绝全篇 `<div>`。

## 3. 审查执行清单 (Checklist)
审查每个文件时，必须进行以下灵魂拷问：
1. **必要性**: 这个模块/组件/函数真的需要存在吗？能不能直接删掉？
2. **复用性**: 这个逻辑是不是在其他地方写过？能否抽离成 Hook/Utils/Service？
3. **精简度**: 100 行的代码能不能用更优雅的 20 行写完？
4. **鲁棒性**: 传入 `null`、`undefined` 或极端异常值，它会炸吗？

## 4. 交付件要求 (Deliverables)

在完成全量代码审查后，必须输出结构化的报告。

- 冗余度分析 (可直接删除的模块/文件)
- 文档缺失列表 (现有 .md 文档需要补充的知识点)

| --- | --- | --- | --- |
